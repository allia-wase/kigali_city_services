# Kigali City Services & Places Directory

A Flutter mobile application that helps Kigali residents locate and navigate to essential public services and leisure locations including hospitals, police stations, libraries, restaurants, cafés, parks, and tourist attractions.

## Features

- **Authentication**: Sign up, log in, and log out using Firebase Authentication (email/password). Email verification is required before accessing the app. New users receive a verification email on sign-up.
- **User Profiles**: Each authenticated user has a profile stored in Firestore (`profiles` collection), linked by UID. Profiles store email and display name.
- **Listings CRUD**: Create, read, update, and delete location listings. Only the listing owner can update or delete their listings. Listings include name, category, address, contact, description, and coordinates.
- **Directory**: Browse all listings with real-time updates from Firestore. Listings are ordered by creation timestamp (newest first).
- **Search & Filter**: Search listings by name and filter by category (Hospital, Police Station, Library, Restaurant, Café, Park, Tourist Attraction).
- **Map View**: Interactive map (OpenStreetMap via flutter_map) displaying all listings with markers. Tap a marker to open the listing detail.
- **Detail Page**: View full listing information with an embedded map and marker. Navigate button opens turn-by-turn directions in Google Maps.
- **My Listings**: View, edit, and delete listings created by the current user.
- **Settings**: User profile display, location notification toggle, theme (light/dark/system), and log out.

## Firestore Database Structure

The app uses two top-level collections in Cloud Firestore.

### Collection: `profiles`

Stores user profile data. Document ID equals the Firebase Auth UID.

| Field        | Type   | Description                          |
|-------------|--------|--------------------------------------|
| `uid`       | string | Firebase Auth user ID                |
| `email`     | string | User email address                   |
| `displayName` | string | Optional display name (nullable)   |

**Security**: Documents are keyed by UID. Clients should only read/write their own profile (e.g. via `profiles/{uid}`).

### Collection: `listings`

Stores location listings. Document ID is auto-generated by Firestore.

| Field      | Type   | Description                                    |
|-----------|--------|------------------------------------------------|
| `name`    | string | Listing name (e.g. "Kigali Central Hospital")  |
| `category`| string | One of: Hospital, Police Station, Library, Restaurant, Café, Park, Tourist Attraction |
| `address` | string | Street address                                 |
| `contact` | string | Phone or contact info                          |
| `description` | string | Free-text description                      |
| `latitude`   | number | Geographic latitude                        |
| `longitude`  | number | Geographic longitude                       |
| `createdBy`  | string | UID of the user who created the listing    |
| `timestamp`  | timestamp | Creation time (for ordering)             |

**Indexes**: A composite index on `(createdBy)` is required for the "My Listings" query. Firestore will prompt you to create it when first run. A composite index on `(timestamp descending)` is used for the main directory stream.

## State Management Approach (Riverpod)

The app uses **flutter_riverpod** for state management. All Firestore operations go through dedicated services; widgets never call Firestore directly.

### Provider Types

- **StreamProvider**: Exposes Firestore streams (auth state, profiles, listings). The UI rebuilds when data changes.
- **StateProvider**: Holds ephemeral UI state (e.g. search query, category filter).
- **Provider**: Derives state from other providers (e.g. `filteredListingsProvider` combines `listingListProvider`, `searchQueryProvider`, and `categoryFilterProvider`).

### Provider Layout

| Provider                | Type          | Source                         | Purpose                               |
|-------------------------|---------------|--------------------------------|---------------------------------------|
| `authStateProvider`     | StreamProvider| AuthService.authStateChanges   | Current Firebase user or null         |
| `userProfileProvider`   | StreamProvider| AuthService.userProfileStream  | Current user's Firestore profile      |
| `listingListProvider`   | StreamProvider| FirestoreService.listingsStream| All listings, real-time               |
| `userListingsProvider`  | StreamProvider| FirestoreService.userListingsStream | Current user's listings          |
| `searchQueryProvider`   | StateProvider | -                              | Search text                           |
| `categoryFilterProvider`| StateProvider | -                              | Selected category or null (All)       |
| `filteredListingsProvider` | Provider   | listingList, search, filter    | Filtered list for Directory           |
| `flutterThemeModeProvider` | StateProvider| -                           | Light, dark, or system theme          |

### Data Flow

1. **Services** (`AuthService`, `FirestoreService`) encapsulate Firebase calls.
2. **Providers** subscribe to services and expose streams/state to the UI.
3. **Screens** use `ref.watch()` for reactive data and `ref.read()` for one-off actions.

No business logic or Firestore calls live in widgets.

## Firebase Setup

1. Create a Firebase project at [Firebase Console](https://console.firebase.google.com).
2. Enable **Authentication** (Email/Password) and **Cloud Firestore**.
3. Add an Android app, download `google-services.json`, place in `android/app/`.
4. Add an iOS app, download `GoogleService-Info.plist`, place in `ios/Runner/`.
5. Run `flutter pub get` and `flutter run`.

See [docs/setup.md](docs/setup.md) for step-by-step instructions.

## Running the App

```bash
flutter pub get
flutter run
```
