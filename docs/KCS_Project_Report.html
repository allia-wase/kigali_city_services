KCS – Kigali City Services & Places Directory

Flutter Mobile Application – Project Report

Version 1.0.0

1. Abstract

KCS (Kigali City Services) is a Flutter mobile application that helps residents of Kigali locate and navigate to essential public services and leisure locations. The app uses Firebase for authentication and Firestore for real-time data, Riverpod for state management, and includes a map view with markers. Users can search, filter by category, create and manage listings, and open turn-by-turn navigation via Google Maps.

2. Introduction

KCS addresses the need for a centralized way to discover services and places in Kigali. The app requires authentication with email verification, stores listings in Firestore, and provides search, category filters, and an interactive map. All Firestore operations flow through a service layer; the UI consumes state via Riverpod providers.

3. Features

  • Authentication (signup, login, logout) with email verification
  • Listings CRUD with name, category, address, contact, description, coordinates
  • Directory with search and category filter; real-time Firestore updates
  • Map view with markers; listing detail with embedded map and Navigate button
  • My Listings with loading/error states during delete
  • Settings: profile, email verification status, theme, notification toggle

6. State Management Workflow

Firestore streams flow into FirestoreService, which exposes listingsStream() and userListingsStream(). Riverpod StreamProviders subscribe to these. filteredListingsProvider derives filtered results from searchQueryProvider and categoryFilterProvider. UI uses ref.watch() for reactivity. No direct Firebase calls in widgets.

7. Reflection – Firebase Integration Errors & Resolutions

Error 1: Firebase initialization failing on app startup

Problem: The app sometimes showed "Firebase Setup Required" or crashed on launch. This occurred when google-services.json (Android) or GoogleService-Info.plist (iOS) was missing or misconfigured.

Resolution: Used flutterfire configure to generate firebase_options.dart and ensure native config files were in the correct paths: android/app/google-services.json and ios/Runner/GoogleService-Info.plist. Also wrapped Firebase.initializeApp() in a try-catch so the app shows a clear error message instead of crashing, and runs in demo mode if Firebase is not configured.

Error 2: Firestore "Missing index" when querying listings

Problem: Querying the listings collection with orderBy('timestamp', descending: true) and where('createdBy', isEqualTo: uid) triggered a Firestore error: "The query requires an index."

Resolution: Opened the link provided in the Firestore error message in the browser, which redirected to the Firebase Console. Created the composite index for the listings collection on createdBy (ascending) and timestamp (descending). After the index was built, the queries ran successfully and Directory and My Listings screens displayed data correctly.

Error 3: Email verification state not refreshing after verification

Problem: After the user verified their email via the link, the app still showed the Verify Email screen until the user manually logged out and logged back in.

Resolution: The authStateProvider (StreamProvider) already listens to authStateChanges(), but emailVerified is a property of the User object that does not update in place. The user must call user.reload() and then user = FirebaseAuth.instance.currentUser to get the updated verification status. Added a "Check verification" or "Refresh" button on the Verify Email screen that calls user.reload() and triggers a rebuild, so the app redirects to HomeShell once verification is detected.

8. Conclusion

KCS meets the assignment requirements: Firebase Auth with email verification, Firestore CRUD for listings, search and category filtering, map integration with markers and navigation, Riverpod state management, and loading/error handling. The Reflection above documents Firebase integration issues and how they were resolved.
